# Alive Domain Docgen

> 개발 프로세스를 바꾸지 않고,
> 커밋이 남긴 흔적을 도메인 지식으로 변환합니다.

---

## 한 줄 요약

커밋을 도메인 학습 이벤트로 승격시켜, 라이브한 도메인 문서를 자동으로 생성하는 시스템

---

## 핵심 개념

> 커밋은 단순한 변경 기록이 아니라, **도메인 이해의 스냅샷**이다.

### 기본 아이디어

- **문서를 쓰게 만드는 것 ❌**
- **이미 하는 행위를 문서로 전환 ✓**

Git 커밋 히스토리를 도메인 타임라인으로 재정의합니다.

## 작동 방식

```
1. 개발자가 평소처럼 커밋
   ↓
2. AI가 변경 내용을 도메인 관점으로 분석
   ↓
3. 도메인 문서가 자동으로 최신화
```

**핵심**: 개발자는 아무것도 바꾸지 않습니다. 백그라운드에서 조용히 작동합니다.

자세한 구조: [시스템 아키텍처](docs/07-architecture.md)

## 6가지 설계 원칙

### 1. AI는 기록자, 개발자가 주체
의도 추론이 필요하면 개발자에게 물어봅니다. 문서 신뢰성을 지킵니다.

### 2. 살아있는 문서
운영에 나간 코드만 문서화합니다. 실험과 실패는 담지 않습니다.

### 3. 책임의 분리
문서는 현재의 도메인 진실만, 과거 이력은 이슈 트래커가 담당합니다.

### 4. 현재 중심
지금 고객이 쓰는 정책, 지금 운영되는 규칙만 담습니다.

### 5. 리팩토링의 길잡이
문서는 설계의 결과물이 아니라 설계를 가능하게 하는 토대입니다.

### 6. 프로세스를 방해하지 않음 (가장 중요)
강제 ❌ 권장 ❌ 게이트 ❌
존재를 느끼지 못하는 백그라운드 도구입니다.

상세: [핵심 설계 원칙](docs/03-design-principles.md)

## 레거시 프로젝트 대응

**완벽한 초기 분석을 포기합니다.**

- 지금 바뀐 코드를 기준으로 점진적 개선
- 초반엔 볼품없어도 괜찮음
- 라이브한 상태로 업데이트되면 자연스럽게 성장

자세히: [레거시 대응 전략](docs/04-legacy-strategy.md)

---

## 리팩토링 재정의

> 리팩토링은 코드 정리가 아니라,
> **도메인을 재해석한 결과**입니다.

커밋은 도메인 이해가 진화한 흔적입니다.

자세히: [리팩토링의 재정의](docs/06-refactoring-redefined.md)

## 기대 효과

- 조직 지식 인프라 구축
