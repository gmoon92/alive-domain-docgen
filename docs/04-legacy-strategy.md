# 레거시 대응 전략: "완벽한 초기 분석"의 포기

## 핵심 원칙

**완벽한 초기 분석을 과감히 포기한다**

## 문제 인식

레거시 프로젝트에서 도메인 경계를 찾는 것은 거의 불가능하다. 전체를 한 번에 학습하려는 시도는 다음 이유로 실패한다:

- 컨텍스트/토큰 비용 폭발
- 도메인 경계의 모호함
- "먼저 전체를 이해해야 한다"는 강박

## 우리의 접근

- **"지금 바뀐 코드"를 기준으로** 도메인 문서를 조금씩, 역방향으로 채워 나감
- 초반엔 볼품없을지라도 프로젝트가 라이브한 상태로 업데이트가 빈번하다면 점진적으로 개선됨
- 도메인은 "변경 이력"에서 드러난다

> 초반 품질은 낮아도 지속적 업데이트로 점진적 개선.
> 문서가 "완성품"일 필요는 없다. 라이브하면 된다.

## 왜 이 방식이 작동하는가

1. **변경이 일어나는 곳이 가장 중요한 도메인**
   - 자주 바뀌는 코드 = 비즈니스 핵심
   - 안 바뀌는 코드 = 일단 신경 쓸 필요 없음

2. **점진적 개선은 린(Lean) 철학과 일치**
   - 완벽한 초기 설계 대신 빠른 피드백
   - 실제 사용을 통한 개선

3. **문서는 살아있어야 한다**
   - "완성된 문서"는 곧 죽은 문서
   - 계속 업데이트되는 문서가 신뢰받는다

## 성공 조건

> 초반엔 볼품없을지라도 프로젝트가 라이브한 상태로 업데이트가 빈번하다면
> 이 프로젝트는 성공할 것이다.
